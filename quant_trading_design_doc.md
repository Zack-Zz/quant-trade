# A股量化交易系统（Python 研究/信号 + Java 执行）落地设计文档（MVP→可扩展）

> 目标：个人自用，偏保守策略；同时支持 **ETF** 与 **个股**；采用“**Python 出信号/模型** + **Java 执行/风控/调度**”的分层架构，先跑通最小闭环，再逐步扩展到稳定实盘。

---

## 1. 背景与目标

### 1.1 你要解决的核心问题
- **策略研究与验证**：快速迭代、回测可信、避免过拟合与数据偏差。
- **交易执行的工程可靠性**：幂等、对账、重启恢复、异常保护、风控熔断。
- **同时支持 ETF 与 个股**：同一套执行框架，不同的策略插件与标的池配置即可复用。

### 1.2 目标边界（MVP明确不做/后做）
**MVP 先做：**
- 日/周/月级别低频策略（保守、成本敏感低）。
- 统一信号协议：**目标仓位（Target Portfolio）**。
- 纸交易（模拟撮合）+ 小资金实盘（券商通道适配）。
- 关键风控：单票上限、换手限制、单日限亏、最大回撤熔断、交易时段限制、A股约束（T+1、停牌、涨跌停保护）。

**后续再做：**
- 高频/分钟级、复杂撮合与盘口。
- 期权/期货对冲、套利类复杂策略。
- 多账户、多券商、多实例高可用集群（个人自用可后置）。
- 可视化大屏（先用报告输出/简单Web面板即可）。

## 2. 总体架构

### 2.1 分层原则
- **Python 层**：研究/回测/纸交易/信号服务（快迭代）。
- **Java 层**：交易执行/风控/调度/对账/恢复（强工程可靠性）。

### 2.2 逻辑架构图（文本版）

```
                ┌─────────────────────────────────────┐
                │            Python：Research          │
                │  - 数据处理/因子/策略/回测/报告       │
                └───────────────┬─────────────────────┘
                                │（生成信号）
                                ▼
                ┌─────────────────────────────────────┐
                │          Python：Signal Service      │
                │  - /signal 输出 Target Portfolio     │
                │  - /health /version /explain         │
                └───────────────┬─────────────────────┘
                                │ HTTP（MVP）
                                ▼
┌───────────────────────────────────────────────────────────────┐
│                   Java：Trade Executor（核心）                 │
│  1) 拉取信号 → 2) 风控评估/降级 → 3) 订单规划 → 4) 下单执行     │
│  5) 订单状态机 → 6) 对账 → 7) 日志/监控/告警 → 8) 重启恢复       │
└───────────────┬───────────────────────┬───────────────────────┘
                │                       │
                ▼                       ▼
     ┌──────────────────┐     ┌──────────────────────────┐
     │ Paper Broker      │     │ Live Broker（券商适配）   │

     └──────────────────┘     └──────────────────────────┘
```

### 2.3 通信方案
- **MVP：HTTP 拉取**
  - Python 提供 `/signal`，Java 定时拉取并执行。
  - 优点：开发最快、调试方便。
- **扩展：消息队列推送（可选）**
  - Python 发 `SignalEvent` 到 MQ；Java 消费执行，天然可重放、解耦更强。

## 3. 关键抽象：统一信号协议（Target Portfolio）

### 3.1 为什么选“目标仓位”而不是“直接下单”
- 保守策略更适合表达“我想持有多少资产”，执行端可以：
  - 做风控降级（降仓/清仓）  
  - 做换手限制、分拆下单、价格保护  
  - 做交易约束处理（涨跌停/停牌/T+1）

### 3.2 Signal JSON Schema（MVP 版本）
```json
{
  "as_of": "2026-03-06T14:55:00+08:00",
  "strategy_id": "etf_trend_v1",
  "universe": ["510300.SH", "515180.SH"],
  "portfolio": {
    "cash_target_pct": 0.10,
    "targets": [
      {"symbol": "510300.SH", "target_pct": 0.60, "confidence": 0.72, "reason": "trend_up"},
      {"symbol": "515180.SH", "target_pct": 0.30, "confidence": 0.61, "reason": "defensive_alloc"}
    ]
  },
  "constraints": {
    "max_turnover_pct": 0.20,
    "max_single_position_pct": 0.10,
    "rebalance": "WEEKLY_CLOSE",
    "min_avg_daily_amount": 200000000,
    "exclude_tags": ["ST", "退市", "停牌"]
  },
  "checksum": "sha256:..."
}
```

### 3.3 字段解释
- `as_of`：信号生成时间（用于幂等与审计）
- `strategy_id`：策略版本（用于风控配置与路由）
- `universe`：本次参与标的池（便于审计/解释）
- `portfolio.cash_target_pct`：目标现金比例
- `targets[]`：目标持仓权重（总和 ≤ 1 - cash）
- `constraints`：建议约束（Java 会二次校验并可覆盖）
- `checksum`：可选，用于防篡改与一致性校验

## 4. 策略分类与支持范围（ETF + 个股）

### 4.1 ETF 策略分类（优先实现）
1) **资产配置 / 再平衡（超保守）**
- 例：股债平衡、风险平价、波动率目标、月度再平衡

2) **趋势跟随（保守/中低频）**
- 例：均线趋势、时间序列动量、突破（低频版）

3) **轮动（中等风险/低频）**
- 例：多ETF动量 TopK + 防御资产切换

### 4.2 个股策略分类（第二阶段实现）
1) **指数增强（最稳的个股方向）**
- 股票池：沪深300/中证500成分股，或成交额 TopN
- 策略：低频动量/趋势 TopK、行业约束/风险暴露控制（可选）

2) **因子选股（依赖基本面数据质量）**
- 价值/质量/低波动/红利等，月/季调仓

> 强烈建议：个股第一版走“指数增强 + 强过滤 + 分散持仓”，避免事件驱动/题材类高波动策略。

## 5. A股交易规则与回测/执行的关键建模

### 5.1 必须建模（否则回测会“骗人”）
- **T+1**：当天买入不可当日卖出（执行端必须强校验）
- **涨跌停不可成交**：至少做保护逻辑（买不到/卖不出要风控降级）
- **停牌不可交易**：持仓处理与信号执行差异
- **交易成本**：佣金、印花税（卖出）、过户费（视市场/券商规则）
- **最小交易单位**：A股通常 100 股一手（ETF也有一手）

### 5.2 执行端保护建议
- 价格保护：限价单（相对最新价±阈值）、不追涨停、不抢跌停
- 失败降级：下单失败/行情异常 → 停止开仓、仅允许减仓
- 风控熔断：单日限亏触发 → 立即停止新订单

## 6. Java 执行层设计（核心可落地）

### 6.1 执行主流程（Pipeline）
1. 拉取信号（SignalClient）
2. 幂等判断（ExecutionLedger：`strategy_id + as_of`）
3. 拉取账户快照（资金、持仓、当日已买入不可卖等）
4. 风控评估（RiskEngine）
5. 订单规划（OrderPlanner：从现有仓位到目标仓位）
6. 下单执行（Broker.execute）
7. 订单状态机（OrderStateMachine）
8. 对账（Reconcile：本地 vs 券商）
9. 记录与报告（TradeJournal）
10. 监控告警（Prometheus/日志/邮件/钉钉/企业微信等）

### 6.2 核心模块
- **SignalClient**：HTTP 拉取 `/signal`
- **RiskEngine**：规则集合（可配置、可组合）
- **OrderPlanner**：
  - 输入：目标权重 + 当前持仓 + 可用现金 + 价格
  - 输出：买卖意图（symbol, side, shares, limitPrice）
- **Broker**：
  - PaperBroker：模拟撮合（用于纸交易与回归测试）
  - LiveBroker：券商适配（下单/撤单/查询/成交回报）
- **ExecutionLedger**：幂等与审计（SQLite/PostgreSQL/本地文件均可）
- **Reconcile**：对账恢复（重启后根据券商状态恢复本地订单/持仓）

### 6.3 风控规则清单（MVP必须）
通用：
- 单票最大仓位 `max_single_position_pct`
- 最大换手 `max_turnover_pct`
- 单日最大亏损 `max_daily_loss_pct`
- 最大回撤熔断 `max_drawdown_pct`
- 交易时段限制（避开集合竞价/尾盘可选）
- 黑名单/白名单（symbol级）

个股加严：
- 剔除 ST/退市/停牌
- 流动性阈值：近20日平均成交额 >= X
- 最小持仓数/最大持仓数（分散）
- 涨跌停策略：不追、不补、触发风险降级

### 6.4 幂等与重启恢复
- **幂等键**：`strategy_id + as_of`（或加上 trading_date）
- 执行记录：每次执行的信号、风控决策、订单计划、订单回报都要落库
- 重启恢复：启动时读取券商未完成订单与当前持仓，修正本地状态

## 7. Python 研究层设计（可快速迭代）

### 7.1 模块职责
- **data/**：行情获取、复权处理、交易日历、标的元数据
- **strategy/**：策略插件（ETF/个股）
- **backtest/**：回测引擎（含交易约束、成本模型）
- **paper/**：模拟盘（可复用回测撮合逻辑）
- **serve/**：信号服务（FastAPI），输出目标仓位 JSON
- **report/**：回测报告与实盘复盘（HTML/Markdown/CSV）

### 7.2 策略插件接口（推荐）
输入：
- 交易日历/价格数据/持仓（可选）/参数

输出：
- `Signal(Target Portfolio)` + `Explain`（原因、关键指标）

示例（伪接口）：
```python
class Strategy:
    def generate_signal(self, ctx) -> Signal:
        ...
```

### 7.3 模型（ML）引入建议
- 第一阶段不强依赖 ML：先用确定性策略（趋势/配置）跑通闭环
- ML 更适合第二阶段做“状态识别/仓位控制”：
  - 预测市场状态：趋势/震荡/高波动
  - 输出风险系数：用于降仓/切换防御资产

## 8. 数据与存储设计

### 8.1 数据类型
- 行情：日K（MVP足够）、后续可扩分钟
- 标的元数据：交易所、最小单位、是否ETF、是否ST、停牌状态等
- 交易日历：A股交易日、节假日
- 基本面：第二阶段（若做因子选股）

### 8.2 存储选型（MVP）
- Python：
  - Parquet/CSV（本地）
  - 或 SQLite（简单可控）
- Java：
  - SQLite（个人单机非常够用）
  - 或 PostgreSQL（更稳健，方便查询与扩展）

### 8.3 关键表（Java ledger）
- `signal_log`：信号原文、checksum、拉取时间
- `risk_decision`：风控结果、降级原因
- `order_plan`：计划订单（意图）
- `orders`：订单状态机
- `fills`：成交明细
- `positions`：持仓快照（每日）
- `pnl_daily`：每日收益/回撤

## 9. 订单规划（Target Weight → Shares）要点

输入：
- 目标权重 `target_pct`
- 总资产 `equity`
- 当前持仓市值/股数
- 可用现金
- 最新价（或计划成交价）
- 交易单位（100股/手）

输出：
- 买卖股数（向下取整到手）
- 换手限制：若超 `max_turnover_pct`，按比例缩放订单
- T+1限制：当日买入的股票不允许卖出（减少卖单或延后）

## 10. 监控、日志、告警（必须做）

MVP 监控指标：
- 信号拉取成功率/延迟
- 订单成功率/失败原因分布
- 当日PnL、累计回撤、触发风控次数
- 行情数据断流次数

告警：
- 信号服务不可用
- 下单失败/接口异常
- 风控熔断触发
- 对账差异（本地与券商不一致）

## 12. 开发路线图（按周划分，确保能落地）

### Week 1：打通研究与信号服务（Python）
- 行情数据拉取与缓存（日K）
- 交易日历
- 一个 ETF 保守策略：趋势/再平衡（输出 Target Portfolio）
- FastAPI `/signal` 与 `/health`
- 回测最小版（含成本、手数、T+1简化）

交付标准：
- 本地能跑回测并输出报告（收益/回撤/换手）
- `/signal` 返回可执行的目标仓位 JSON

### Week 2：Java 执行闭环（先 PaperBroker）
- SignalClient（HTTP）
- ExecutionLedger（SQLite）
- RiskEngine（最小规则集）
- OrderPlanner（目标仓位→买卖手数）
- PaperBroker（模拟成交，回归测试）
- 日志与简易报告输出

交付标准：
- 能从 Python 拉信号 → 风控 → 生成订单 → 模拟执行 → 生成当日持仓与PnL

### Week 3：A股约束完善 + 回测对齐
- 停牌/涨跌停/手数/成本更精确建模
- T+1 执行约束与测试用例
- 风控完善：单日限亏、最大回撤熔断、换手缩放

交付标准：
- 回测与纸交易在同一策略下表现一致性提升（误差可解释）

### Week 4：接入实盘（LiveBroker）+ 小资金试运行
- 券商接口适配（你选择通道后实现）
- 对账与重启恢复
- 告警接入（邮件/IM任选）
- 小资金、单策略、少标的灰度

交付标准：
- 实盘稳定运行（至少连续两周）
- 无重复下单、可重启恢复、风控可触发且可验证

### Week 5+：扩策略（个股指数增强 + ETF轮动）与可视化
- 个股池：沪深300成分股（或成交额TopN）
- 个股 TopK 动量（周/月调仓）
- 简单Web面板/报告自动化
